Things to do

Note the following refinement to the PR workflow

You are no longer required to add a Ready for review comment to every PR (if there is no comment, we assume the PR is ready for review).
Instead, if the PR is not ready for review yet, add a comment Work in progress. When the PR is ready, add a comment Ready for review.

Submitting clean pull requests

You are reminded to keep PRs independent of each other. Always switch to the master branch before creating a new branch to avoid commits from one PR being included in other PRs. Before submitting a PR, ensure that the code is clean, i.e. compliance with coding style and no commented out code. Moreover, ensure that tests and documentation have been updated where necessary.

T7A1. Reach V0.1

Work as a team to deliver V0.1. Note the following:
  • Refer to our instructions in the handbook to find possible workflow options.
  • Divide components (GUI, Model, etc.) among team members. Learn the code of your component. In V0.1 (and V0.2), try to limit your work to your component only.
  • Change code in small steps, run tests after each step, commit each step, and merge often.
Submission:
As your project starts to diverge from AddressBook codebase, it is no longer meaningful to use PRs to indicate progress.
Instead, push all the relevant changes to the main branch of your repo once it reaches V0.1 requirements. Remember to use git tag to annotate that commit as V0.1.
